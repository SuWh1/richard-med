## 1. Главная идея за 10 секунд

**Richard Med — Aviasales для медицинских услуг в Казахстане.**

Пользователь вводит одну услугу: “терапевт”, “УЗИ”, “глюкоза”. Система показывает реальные
цены клиник, источник цены, дату обновления, карту, врача/клинику, отзывы, сравнение и
возможность сохранить услугу в личный кабинет для отслеживания цены.

**Тезис, который надо повторять:**
большинство команд делают “парсер + таблицу”. Мы сделали **доверенный слой цен**:
каждая цена проверяема, нормализована, свежая и привязана к источнику.

## 4. Архитектура простыми словами

Система состоит из трёх больших слоёв.

### Слой 1. Сбор данных

Парсеры работают отдельно от пользователя: вручную из админки или по расписанию.

Контракт источника:

```text
fetch(city) -> raw documents
parse(raw_doc) -> raw price rows
clean(raw_item) -> cleaned raw row
```

Важно: адаптеры **не пишут в БД сами**. Они только получают и разбирают данные. Persistence,
нормализация, дедупликация, история цен и ошибки живут в общем pipeline.

Источники:

- `kdl_olymp` — лаборатории, много городов, плюс филиалы для карты.
- `doq` — врачи, клиники, цены приёма, врачебные карточки.
- `invitro` — лабораторные услуги.
- `helix` — лабораторные услуги.

### Слой 2. Нормализация

Проблема: одна услуга называется по-разному: “ОАК”, “CBC”, “клинический анализ крови”.

Каскад:

1. Очистка строки: нижний регистр, `ё -> е`, латиница/кириллица, лишняя пунктуация.
2. Exact match по каталогу.
3. Alias match по `service_aliases`.
4. Fuzzy match через RapidFuzz.
5. Semantic match через embedding — осторожно: не публикуем автоматически сомнительные совпадения.
6. Если уверенности нет — `unmatched_services`, очередь на проверку.

Фраза для жюри:  
**“Мы не притворяемся, что ИИ идеально понял медицину. Всё уверенное публикуется, всё серое уходит в очередь.”**

### Слой 3. Пользовательский поиск

В момент поиска нет live scraping.

```text
React ResultsPage
  -> GET /api/v1/search?q=...&city=...
  -> resolve_query()
  -> prices_for_service()
  -> cards + suggestions + other_cities
```

Бэкенд:

- резолвит текст в `service_id`;
- берёт только `clinic_service_prices.is_active = true`;
- по умолчанию скрывает данные старше 30 дней;
- собирает карточку: цена, клиника, филиал, врач, рейтинг, свежесть, источник, raw name, confidence;
- если в выбранном городе пусто, показывает города, где по этой услуге есть свежие цены.

## 5. Путь одной цены

Выучи это как главный технический ответ:

```text
сайт / API источника
  -> adapter.fetch()
  -> raw_documents: HTML/JSON + source_url + content_hash + fetched_at
  -> adapter.parse()
  -> raw_price_items: сырое название, сырая цена, metadata
  -> ServiceMatcher: exact / alias / fuzzy / semantic / unmatched
  -> clinic_service_prices: активная цена, service_id, clinic_id, city, parsed_at, source_url
  -> Price Passport в интерфейсе
```

Если цена изменилась:

```text
старый clinic_service_prices.is_active = false
новая цена становится active
price_history получает old_price / new_price / percent_change
```

Если источник не вернул часть старых услуг:

- данные не удаляются физически;
- активные цены источника могут деактивироваться;
- raw evidence остаётся.

## 6. Что именно пользователь видит

### Поиск

- `/search`
- автодополнение через `/services`;
- карточки с “ценой-героем”;
- сортировки;
- фильтры;
- подсказка по другим городам, если в текущем городе нет свежих цен.

### Карта

- `/search/map`;
- цена сети может быть city-wide, а карта “разворачивает” её по филиалам;
- пины показывают цену;
- список и карта синхронизированы.

### Паспорт цены

Показывает почему цене можно доверять:

- источник;
- дата парсинга;
- сырое название услуги;
- нормализованная услуга;
- confidence и method;
- content hash;
- URL источника.

### Сравнение

- `/search/compare`;
- сравнение выбранных клиник по одной услуге;
- `/search/compare/insight` даёт AI-обзор по отзывам, если настроен Gemini API;
- AI не придумывает цены, он читает уже сохранённые отзывы/рейтинги из нашей БД.

### Личный кабинет

Кабинет теперь не placeholder:

- сохранённые услуги;
- история поиска;
- in-app уведомления о смене цены;
- save/unsave из результатов поиска.

Логика price watch:

```text
user_saved_services:
  user_id, service_id, optional clinic_id, city
  baseline_min_price
  last_seen_min_price
  notify_enabled

notification appears when:
  notify_enabled = true
  current fresh price != last_seen_min_price
```

Email в этой версии нет. Это надо говорить честно: **“в этом pass уведомления внутри продукта,
без внешнего email-провайдера.”**

### Админка

Админка нужна не для пациента, а для доказательства качества данных:

- Source Health;
- список parse runs;
- каталог услуг;
- очередь unmatched;
- ручной запуск парсера.

## 7. Как объяснить соответствие кейсу

| Требование кейса | Порог | У нас | Ответ |
|---|---:|---:|---|
| Источники | ≥3 | 4 | превышаем |
| Услуги | ≥100 | 34k активных цен, 3 534 рабочих услуги | превышаем |
| Нормализованные позиции | ≥50 | 1 281 базовых + авто-доращивание | превышаем |
| Raw vs normalized layer | обязательно | `raw_documents`, `raw_price_items`, `clinic_service_prices` | закрыто |
| Manual + cron parsing | обязательно | admin trigger + APScheduler | закрыто |
| Ошибки источников | обязательно | `parse_runs.errors`, per source/city isolation | закрыто |
| Свежесть | ≥1 раз/день | scheduler + parsed_at + freshness badges | закрыто |
| Не показывать >30 дней как current | обязательно | stale hidden by default | закрыто |
| Поиск ≤3 сек | обязательно | DB-only, обычно <1 сек | закрыто |
| Добавлять источники без переписывания ядра | обязательно | adapter registry | закрыто |
| Public data, no PII | обязательно | публичные источники, 2GIS не скрапим как listings | закрыто |

## 8. Почему это технически сильнее “обычного парсера”

Обычный парсер:

```text
сайт -> таблица цен
```

Richard Med:

```text
источник -> raw evidence -> raw rows -> normalization -> active prices
  -> freshness -> passport -> search -> map -> compare -> cabinet watch
```

Сильные инженерные решения:

- source adapters изолированы;
- pipeline хранит raw evidence;
- нормализация отделена от парсинга;
- пользовательский поиск не зависит от сайтов клиник;
- stale данные не смешиваются с current;
- unmatched queue защищает от ложных совпадений;
- price history даёт основу для price watch;
- API типизирован через Pydantic и фронтовые TS-типы;
- auth через JWT Bearer;
- 49 тест-файлов покрывают backend и frontend сценарии.

## 10. Готовые ответы на вопросы жюри

### “Это легально?”

Мы работаем с публичными данными, не трогаем закрытые кабинеты, не берём медицинские карты,
не собираем patient PII. 2GIS listings мы не скрапим; отзывы/рейтинги обновляются отдельным
background sync через доступный API/HTTP слой, а маршруты — как deeplink.

### “Почему поиск быстрый?”

Потому что в поиске нет скрапинга. Парсинг и нормализация проходят заранее. В момент запроса
мы делаем lookup по PostgreSQL: service resolution + active prices.

### “Как вы нормализуете разные названия?”

Каскад: canonical clean, exact, aliases, fuzzy RapidFuzz, semantic suggestion, unmatched queue.
Автопубликуем только уверенные совпадения. Сомнительное не попадает в результат как current price.

### “Что если сайт клиники упал?”

Один источник не ломает остальные. Pipeline идёт source/city изолированно, ошибки пишутся в
`parse_runs`, старые данные остаются с датой свежести. Пользовательский поиск продолжает
работать по БД.

### “Как добавить новый источник?”

Добавляется один adapter file с `fetch/parse/clean/identity`. Registry подключает источник,
а общий pipeline уже умеет raw evidence, normalization, dedupe, price history и health logs.

### “Где тут AI?”

AI не является источником цен. Он используется аккуратно:

- semantic matching как вспомогательный слой нормализации;
- AI insight в сравнении клиник на основе уже сохранённых отзывов.

Цены всегда source-backed и проверяемы.

### “Почему у вас 3 534 услуги, если каталог 1 281?”

1 281 — официальный каталог кейса. Но реальные источники содержат больше формулировок.
Мы не выбрасываем их: уверенные позиции нормализуем, новые добавляем через auto-grown catalog,
а сомнительные держим в unmatched queue.

### “Почему в кабинете нет email?”

В этом pass мы сделали account-backed in-app price watch: сохранение услуги, baseline,
last-seen price, уведомление в кабинете. Email/provider — следующий слой, но продуктовая
логика отслеживания уже есть.

### “Чем вы лучше Kaspi/2GIS/сайтов клиник?”

Сайты клиник дают только свой прайс. 2GIS помогает найти клинику, но не нормализует цены
по одной медицинской услуге. Richard Med агрегирует, нормализует, сравнивает и показывает
доказательство цены.

### “Это медицинская рекомендация?”

Нет. Мы не говорим где лечиться и не ставим диагнозы. Мы даём справочную информацию о ценах,
источниках, свежести, отзывах и маршруте. Медицинское решение остаётся за пациентом и врачом.

### “Как монетизировать?”

B2C доверенный поиск даёт трафик. B2B слой: аналитика цен для клиник, visibility placement,
market intelligence для страховых и госструктур, price index по городам и категориям.

### “Какая главная защита от конкурентов?”

Не сам UI, а data moat: raw evidence, нормализация, история цен, source health, очищенный
каталог и операционный pipeline. Это сложно быстро повторить без качества данных.

## 11. Что нельзя обещать лишнего

- Не говори “мы скрапим live при поиске”. Правильно: **“парсинг офлайн, поиск по БД.”**
- Не говори “ИИ сам определяет цены”. Правильно: **“цены только из источников; AI помогает в обзоре/сопоставлении.”**
- Не говори “email alerts уже отправляются”. Правильно: **“in-app price watch готов, email — следующий слой.”**
- Не говори “все медицинские услуги Казахстана покрыты”. Правильно: **“MVP покрывает 4 источника и 17 городов в текущей БД.”**
- Не стартуй демо с пустого запроса. Используй “терапевт” или “УЗИ”.
- Не спорь про идеальность данных. Правильно: **“мы показываем свежесть и источник, а не скрываем неопределённость.”**

## 12. Быстрый technical deep dive

### Backend

- FastAPI, SQLAlchemy 2.0, Pydantic.
- PostgreSQL 16 + pgvector + trigram/full-text.
- Основные endpoints:
  - `/services` autocomplete;
  - `/search`, `/search/map`, `/search/compare`, `/search/compare/insight`;
  - `/clinics`, `/clinics/{id}`, `/clinics/{id}/services`, reviews;
  - `/doctors/{id}`;
  - `/analytics/overview`, `/analytics/price-stats`;
  - `/cabinet`;
  - `/admin/source-health`, `/admin/parsers/run`, `/admin/unmatched`, `/admin/services`;
  - `/auth/signup`, `/auth/login`, `/auth/me`.

### Frontend

- React 19, TypeScript, Vite, Tailwind.
- Основные страницы:
  - Landing;
  - Results;
  - Clinic Directory / Clinic Detail;
  - Doctor Page;
  - Compare;
  - Analytics;
  - Cabinet;
  - Admin Dashboard;
  - Login/Signup.

### Auth

- JWT Bearer.
- `/cabinet` требует signed-in user.
- `/dashboard` требует admin role.

### Freshness rule

- `fresh`: до 7 дней.
- `recent`: до 30 дней.
- `stale`: старше 30 дней.
- По умолчанию stale не показывается как current.
